# Code Review 约束

## 目标
- Code review 的首要目标是让仓库整体代码健康度持续改善, 而不是追求局部“完美”后再合入.
- reviewer 应优先关注正确性, 回归风险, 可维护性, 架构一致性与测试有效性, 不要把主要精力放在格式化或纯个人偏好上.
- 对于纯建议项, 应明确标注为非阻塞意见; 对于会降低代码健康度的问题, 应明确阻止合入.

## 默认流程
- 只要任务是 code review, PR review, pre-merge check, 代码质量检查或“看看这段代码有没有问题”, 默认主动使用 `pragmatic-clean-code-reviewer`.
- 若用户没有给出级别校准信息, 按该 skill 的默认规则使用 `L3 Team`.
- review 时先看 diff 与直接上下文, 再按需读取相关模块; 不要无边界扩散审查范围.
- review 输出优先给出 findings, 按严重性排序; 先说必须修的问题, 再说应修的问题.

## 仓库基础门槛
- 在使用更高阶 clean code 规则前, 先判断是否满足 `note_for_agents/coding_conventions.md` 的基础要求.
- 命名, `unsafe` 使用, `Result` 边界, 配置加载方式与注释风格若已违反仓库硬约束, 应直接作为 review finding 提出.
- 对 Rust 习惯用法的判断, 以本仓库现有模式, workspace 结构和模块职责为准, 不要机械套用面向对象语言规则.
- 能由 `rustfmt`, `clippy`, 编译器或现有自动化稳定捕获的问题, 不应作为 review 主要内容, 除非它掩盖了更深层的逻辑或设计问题.

## 注释审查
- 注释应解释设计意图, 背景约束, 不变量, 兼容性理由或为何不能采用更直接的实现.
- 下列注释默认视为废注释, 除非它们承载额外上下文:
  - 逐字复述代码表面行为的注释.
  - 把变量名, 分支名或函数调用改写成自然语言的注释.
  - 与当前实现不一致的注释.
  - 试图为晦涩实现“补课”, 但没有推动代码或接口本身变得更清晰的注释.
- 如果 reviewer 在评审里要求作者解释一段难懂逻辑, 默认期望结果应是重写代码, 抽取命名更好的结构, 或补充面向未来读者的必要注释, 而不是只在 review 对话里解释清楚.

## 测试审查
- 测试必须证明一个可能被破坏的行为, 契约或回归风险; 不能只证明“代码能编译”或“按当前实现再执行一遍”.
- 下列测试默认视为废测试, 应在 review 中指出:
  - 只验证编译一定成立的类型拼装, 构造或 getter/setter 流程, 且没有行为分支, 边界条件或契约约束.
  - 断言与实现完全同构, 只是把被测逻辑复制到测试里重新算一遍.
  - 没有可观察行为约束, 只覆盖 happy path, 且即使实现出现常见回归也不会失败.
  - 依赖大量 mock 或固定脚手架, 但没有验证真实业务不变量.
- 好测试应至少命中以下之一: 边界条件, 错误传播, 状态转换, 对外契约, 历史回归, 跨模块集成风险.
- 若新增测试只是为了“让覆盖率更好看”, 但不能提升回归捕获能力, 不应视为充分测试.

## Reviewer 行为
- 评论代码, 不评论作者; 措辞保持直接, 清晰, 尊重.
- 提出问题时尽量说明原因, 指出它为什么影响代码健康度, 正确性或可维护性.
- 在“指出问题”与“直接给方案”之间保持平衡: reviewer 不必代写实现, 但应给出足够具体的修复方向.
- 纯教育性或非阻塞建议应明确标成 `Nit`/`Optional`/`FYI` 一类, 避免作者把所有评论都理解为必须修改.
- 如果作者能基于事实, 数据或稳定的工程原则证明另一种实现同样合理, reviewer 应接受作者方案, 不要把个人偏好包装成硬性标准.

## 结论标准
- 只要改动整体上明确提升了系统代码健康度, 即使仍不完美, reviewer 也应倾向于批准.
- 会降低整体代码健康度的改动不应批准, 即使问题分散且“每条都不大”.
- 当 review 结论依赖风险判断时, 优先级顺序是: 安全性 > 正确性 > 回归风险 > 设计一致性 > 风格与可读性细节.
